一、先定“每个小节只保留一个主脊梁”

每个小小节只留三层：

1 篇锚点论文：最早或最有代表性，负责定义这条线

2–4 篇补强论文：分别补不同方向，不要重复

1 句趋势总结：把剩下没写进去的工作打包掉

这样一个小小节通常控制在 4–6 篇 最舒服，再多就会开始挤。

二、按“贡献维度”压，而不是按年份压

不要简单按新旧删，而要问：
这篇是不是提供了新的维度？

如果两篇都在做同一件事，就只留一篇更强的。

你这章里最适合用的维度

对于 Automatic Data Generation，我建议用这几个维度筛：

scale：是否把生成规模显著做大

semantics：是否引入任务/语言可控性

physics：是否引入物理可交互性

grounding：是否锚定真实世界或具体 embodiment

automation：是否形成自动闭环

generality：是否支持更多任务/更多机器人形态

一篇论文如果只是在已有维度上“再做一遍”，就可以删。

三、每个小节内部做“去同类项”
1. Environment and Scene Synthesis

这里最容易重复。

你其实不需要同时细写很多“能自动生成场景”的论文。
更好的压法是：

ProcTHOR：代表 scale

Holodeck：代表 semantic controllability

PhyScene：代表 physical realism

GAIA：代表 real-context grounding

然后：

RoboGen、RoboCasa、GRUtopia、BVS 作为补充或一句带过

也就是说，这一节正文里真正展开的可以只有 4 篇。

2. Task, Demonstration, and Trajectory Synthesis

这一节最容易爆炸，因为你现在塞了很多不同机器人形态的生成工作。

压缩方法是分成三类，每类只留代表：

seed-demo expansion：
留 MimicGen

morphology-aware synthesis：
留 MoMaGen 或 HumanoidGen，二选一，看你综述更偏 mobile manipulator 还是 humanoid

simulator-light / 3D-based synthesis：
留 RoboSplat 或 R2R2R，二选一

强调 3D representation → RoboSplat

强调从真实输入直接扩展 → R2R2R

然后把：

SkillMimicGen

UltraDexGrasp

OmniRetarget

DemoGen

Tether

InternData-A1

作为补充池，不一定都写进主文。

这一节压完后，主干只需要 4–5 篇。

3. Generative World Models for Embodied Data Generation

这一节也容易堆。

建议只保留四种角色：

通用 platform / foundation model：Cosmos

robot interaction–specific predictor：IRASim

compositional manipulation imagination：DreMa 或 RoboDreamer，二选一

world model as environment / downstream usage：WorldGym

然后：

iVideoGPT

AnchorDream

DreamGen

DreamDojo

EnerVerse

作为补充句或表格项。

这一节正文同样压到 4–5 篇 就够了。

四、遇到边界型论文时，只给它一个“主归属”

你现在内容变多的一个核心原因，是同一篇论文在多个地方都想写。

压缩时必须狠一点：

RoboGen 主归属给 Environment and Scene Synthesis

DreamGen 主归属给 Generative World Models 或 Synthetic Data Generation，二选一

Tether 主归属给 Task / Trajectory Synthesis

AutoRT 主归属给 Physical World Data Acquisition

IntervenGen / PLD 主归属给 Data Refinement

别让一篇论文在三处都占主句。
最多在别处一句 “Related efforts include …” 带过去。

五、正文写“代表作”，表格放“全景池”

这是最实用的压缩方式。

你完全没必要把所有找到的论文都塞进段落里。
可以这样分工：

正文：只写代表作和逻辑推进

表格：保留更全的文献池

比如正文里只留 12–15 篇主干，
但表格里可以放 25–35 篇。

这样既不会显得漏文献，也不会把正文写成目录。

六、压缩时的优先级顺序

如果你必须删，我建议按这个顺序删：

先删

功能和别篇高度重复的

位置尴尬的边界型论文

你没法一两句话说清贡献的

没有改变叙事结构、只是“又做了一版”的

后删

代表某个关键新维度的

代表一个新趋势的

能帮你搭建段落逻辑桥梁的

七、给你一版可执行的压缩目标

如果是 Automatic Data Generation 这一章，我建议压到：

正文主干：12–14 篇

Environment and Scene Synthesis

ProcTHOR

Holodeck

PhyScene

GAIA

Task, Demonstration, and Trajectory Synthesis

MimicGen

MoMaGen 或 HumanoidGen

RoboSplat 或 R2R2R

Tether

Generative World Models

IRASim

DreamGen 或 WorldGym

其余放补充

BVS

这样正文会清爽很多，但文献池并没有真的丢。
