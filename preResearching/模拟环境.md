1) Isaac 段后半：主线里真正稳的

DexGraspNet 可以留，而且你现在的数字终于对了：公开项目页和论文都写的是 1.32 million grasps across 5,355 objects，并且这些 grasps 由 Isaac Gym 验证。(github.com
)

FurnitureSim + Factory 也可以留。FurnitureBench 官方明确写了 FurnitureSim is based on Isaac Gym and Factory，所以你把它放在 Isaac 段里是合理的。(clvrai.github.io
)

BiDexHands 也能留。它公开写得很清楚：built in Isaac Gym，支持 thousands of parallel environments，而且 30,000–40,000 FPS 量级是它的核心卖点之一。(arxiv.org
)

这三项都属于：
Isaac 的 GPU-native 并行仿真能力，如何直接转化为大规模 dexterous / assembly 数据与训练支撑。

2) Isaac 段后半：能留，但必须降权写

Newton Solver / NVIDIA Warp 这句现在写得过重。
能核实的是：Newton 目前在 Isaac Sim 文档里被描述为 GPU-accelerated, extensible, and differentiable physics engine，而 Warp 也确实是 differentiable GPU kernel framework。(docs.isaacsim.omniverse.nvidia.com
)
但你现在这句里最危险的是：

“sub-millimeter contact fidelity”：我没看到足够硬的官方来源支撑这句

“supporting precision-critical benchmarks such as IndustReal”：这会让人读成 Newton/Warp 直接支撑了 IndustReal，这层关系并不稳

所以这里最稳的改法是：

保留 Newton/Warp 作为 backend refinement / differentiable optimization tooling

删掉或弱化 “sub-millimeter” 这种具体精度 claim

不要把 IndustReal 写成 Newton/Warp 的直接下游成果

换句话说，这里该写成：

Newton and Warp extend the Isaac stack toward differentiable and gradient-based optimization, complementing precision-critical simulation workflows.

这样就够了。(developer.nvidia.com
)

3) Isaac 段后半：这里有一个明确的分类错误

MimicGen 不该放在 Isaac 段主线上。
这个是最清楚的错位之一。MimicGen 官方仓库和环境说明都写得很明确：它的数据生成环境是 built on robosuite，不是 Isaac 生态主线。(github.com
)
所以如果你把它写成“Isaac ecosystem scales training corpora”的主例子，这在分类上是不对的。

它更合适的去处是：

MuJoCo / robosuite 那条线

或者更高层的 “data generation systems” 小结

结论：MimicGen 应该移走。 (proceedings.mlr.press
)

4) 另一个明确的分类错误

HumanoidBench 不该挂在 Isaac 这句后面。
HumanoidBench 官方站点明确写了：它的 simulation environment uses the MuJoCo physics engine。(humanoid-bench.github.io
)
所以你现在这句：

ExBody and OmniH2O ... evaluated via HumanoidBench

会在读者那里形成一种“这一整条是 Isaac humanoid line”的印象，但 HumanoidBench 本身并不是 Isaac benchmark。这就是明显的归属错误。(humanoid-bench.github.io
)

更稳的处理是：

ExBody / OmniH2O 如果你一定要留，可以把它们当成 humanoid retargeting examples

HumanoidBench 要么移出去单讲 humanoid benchmark，要么直接删掉这句

如果你现在是为了收口，我会建议：

这一整句都删。

5) “Beyond the three core ecosystems” 这段：大方向对，但有几处需要修
这一段里基本正确的

RLBench 属于 CoppeliaSim，而且公开论文就是 100 tasks 量级。(arxiv.org
)
Ravens 属于 PyBullet，这个也对。(pypi.org
)
PerAct2 扩展 RLBench 到 bimanual manipulation，这也对。(arxiv.org
)
Colosseum built on RLBench，并且强调 OOD/generalization perturbations，这也对。(roboticsproceedings.org
)
GEM-Bench 是 language-conditioned robotic manipulation generalization benchmark，这也成立。(arxiv.org
)

这里最该修的一处

HomeRobot 不只是 benchmark。
公开论文和项目页都把它定义成 Open-Vocabulary Mobile Manipulation benchmark + simulation + real-world software stack。(arxiv.org
)
所以你现在把它和 ALFRED/TEACh/OpenEQA 并列成“benchmark”一串，会有点压平了它。

另一处不稳

EmbodiedBench 那句别写 “across AI2-THOR, Habitat, and Minecraft”。
我没查到它官方明确主打 Minecraft；公开介绍更稳定的说法是：它有四个环境/套件，包括 EB-Habitat、EB-ALFRED、EB-Navigation、EB-Manipulation。(openreview.net
)
所以你那句里的 Minecraft 至少在当前公开材料下是不稳的，建议删掉。

6) 最后一段 “frontier of simulation ... Genesis” ：方向对，但也有边界问题

SoftGym、DeformableRavens、FluidLab、DiffSkill、DexDeform 这些放在一起讲“soft / deformable / fluid”这条 frontier，是成立的。
Genesis 公开材料也确实强调自己整合了 MPM、FEM、PBD、SPH/Stable Fluid 这类多种 solver，并支持 rigid / soft / fluid 等多材料耦合。(github.com
)

但这里也有两个要注意的点：

第一，Genesis 这一句更像“新统一引擎宣言”，不是已经成熟的 manipulation benchmark 生态。
所以它可以作为收尾展望，但不要写得像 MuJoCo / SAPIEN / Isaac 那样已经形成稳定生态。

第二，
“natively capturing the interplay among rigid, soft, and fluid bodies for scalable multi-physics data synthesis”
这个方向对，但也偏强。Genesis 官方更稳的说法是“integrates diverse physics solvers and supports coupling across materials”，你现在这句已经有点替它做更大的总结了。
